# AGENTS.md

## 适用范围
- 本文件适用于 `ai-requirement-assistant` 应用。
- 当现有代码已经形成稳定模式时，优先遵循项目内一致性，而不是套用通用最佳实践。

## 项目目标
- 这是一个 AI 需求助手原型项目。
- 优先保证用户流程清晰、迭代速度快、问题容易排查。
- 所有改动应尽量让 `输入 -> API -> 流式结果` 这条主链路保持清晰、易观察、易调试。

## 技术栈偏好
- 使用 Next.js App Router + TypeScript。
- 优先使用 React 函数组件。
- 优先使用 Tailwind CSS 编写样式。
- 优先使用 Vercel AI SDK 处理流式 AI 交互。
- 接入 Packycode 或类似网关时，优先使用 OpenAI-compatible provider。

## UI 库约定
- 默认允许使用：`@ai-sdk/react`、`@ai-sdk/openai-compatible`、`lucide-react`。
- 如果要引入ui库,请说明其效果优势与你想达成的效果.
- 除非确实解决明显问题，否则不要主动引入新的 UI 库。
- 未经明确要求，不要引入Material UI这类重量级 UI 框架。

## 页面风格
- 页面风格保持干净、克制、偏产品化。
- 但是要与市面上明显的ai制式网站做出区分度,不要"主流审美",不要"千篇一律".
- 默认使用浅色主题，除非任务明确要求深色主题。
- 颜色使用低饱和、稳定的产品配色，避免夸张渐变、霓虹色和强视觉噪音。
- 保持足够留白，重视可读性和信息层级。
- 优先做实用型产品界面，而不是偏宣传页的视觉效果。


## 组件设计原则
- 优先先拆小型展示组件，再考虑抽离复杂抽象。
- 不要过度设计。

## AI / API 约定
- API 集成要保留最小但有效的日志，便于调试。
- 不要在源码里硬编码 API Key，统一使用环境变量。

## 错误处理
- 面向用户的错误提示要尽量直白易懂。
- 面向开发排查的信息优先打印到服务端日志。
- 对 `PACKYCODE_API_KEY`、`PACKYCODE_BASE_URL` 这类关键配置缺失要尽早失败。

## 文档要求
- 遇到不算简单的调试、集成、架构调整时，要在 `docs/` 中新增或更新文档。
- 项目知识沉淀优先使用简洁中文文档。
- Troubleshooting 文档尽量按“现象 -> 原因 -> 修复 -> 预防”组织。

## 修改代码时的偏好
- 尽量保持现有命名和目录习惯，除非说明变更理由。
- 解决当前任务时，避免顺手做无关的大重构。
- 如果当前文件里存在很小且安全的清理项，例如无用 import、明显 warning，可以顺手处理。
- 注释保持简短，只写对理解非直观逻辑确实有帮助的内容。

## 验证与检查
- 做了有意义的代码改动后，优先运行 `npm run lint`。
- 如果改动涉及类型、路由、接口边界或集成逻辑，补跑 `npm run build`。
- 如果改动涉及流式行为，除了页面表现，还要同时检查服务端终端日志。

## 沟通偏好
- 解释修改时优先讲清“为什么这样改”和“会影响什么”，不要只讲概念。
- 如果有多种实现方案，优先推荐最简单、且能支持后续聊天组件扩展的方案。
- 调试前后端流式问题时，要明确指出是否存在协议不匹配。
- 由于是学习项目,要照顾到对代码的不熟悉,尽可能提供对代码修改处的解析.
